lets plan out the project, its called HydraDataHive. Its a light weight data replication service with 3 main components. 

1) neigbors
2) data
3) seeding

Hydra App.
it will run in a container, when the container starts with -e MASTER=TRUE it will seed data with a welcome.md and wait for neigbors to send hello's by default a hydra cluster is open / public and require no auth, but new neigbors is set to manual approve by default with a -e NEIGBORS=MANUAL or -e NEIGBORS=AUTO, there is a table with IP,FALSE,LAST_ONLINE or IP,TRUE,LAST_ONLINE for validated neighbors. environment variables will be used to set the master node and the neighbor approval process. the app will have a web interface to view the data store, neighbors, and seeding status. the app will also have a command line interface for advanced users to manage the cluster. environment variables can be provided via .env

Data:
the app has a input folder for new data to be added, it has a data store, and a data out folder. data input will store the file meta data like name, date create, size, extension, mime type, size,uploading server, and checksum in a sqlite db. the file is then split into 64mb base64 chunks and stored in the datastore folder with a subfolder  [id] matching the sqlite record id. The app will have a cron job that runs every 5 minutes to check for new data in the input folder, and if there is new data it will process it and add it to the datastore and then remove the original file.

Neighbors:
Neighbors will send a hello message to the master node when they start up. The master node will respond 
with a list of neighbors and their statuses regardless of approved or not. If a neighbor is set to manual approval, the master node will wait for an admin to approve or reject the neighbor before syncing data from the neighbor using seeded data, the neighbor will be able to download the masters seeded data as soon as the hello handshake completes. Approved neighbors will be able to seed data to the master node and receive updates on new data added to the master datastore. neighbors will also have a cron job that runs every 5 minutes to check for new data in the datastore and sync it with the master node if they are approved. every neighbor will allow other neighbors to connect and sync data with them, creating a mesh network of data replication. during the 5min heartbeat with their master, any new neighbors will be reported to the master,the master will try to add the new neighbors that have joined the network as a neighbor and send hello messages to attempt a direct connection with them.

seeding:
Seeding is the process of distributing data from the approved neighbors to the master node. When a neighbor is approved, it will start seeding its data to the master node. The master node will then distribute this data to all other approved neighbors in the network. This creates a decentralized data replication system where each neighbor can contribute to the overall data pool. The seeding process will also include a mechanism for handling conflicts, such as when two neighbors have different versions of the same file. In such cases, the master node will determine which version is the most recent and keep that version while discarding the older one. This ensures that the data in the network remains consistent and up-to-date across all neighbors. Additionally, the seeding process will also include a mechanism for handling data integrity, such as using checksums to verify that the data being seeded is not corrupted during transmission. Overall, the seeding process is a crucial component of the Hydra app as it allows for efficient and reliable data replication across the network of neighbors.

Node: a node will always download all the data from the master node when it first connects, and then it will only download new or updated data from the master node during subsequent syncs. This ensures that all nodes in the network have a complete and up-to-date copy of the data. the node will optionally sync with other approved neighbors to further distribute the data and reduce the load on the master node. The node will also have a mechanism for handling conflicts, such as when two neighbors have different versions of the same file. In such cases, the node will determine which version is the most recent and keep that version while discarding the older one. This ensures that the data in the network remains consistent and up-to-date across all neighbors. Additionally, the node will also have a mechanism for handling data integrity, such as using checksums to verify that the data being synced is not corrupted during transmission. Overall, the node is a crucial component of the Hydra app as it allows for efficient and reliable data replication across the network of neighbors.
